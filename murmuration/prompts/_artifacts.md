# Local-install artifact scaffolds

> Sub-prompt of the unified `murmuration` skill. Read this when
> `prompts/recommend.md`'s `install` move emits a local artifact —
> cron entry, launchd plist, GH workflow, or gstack skill. The
> scaffolds here are the canonical structure for each emit format.
> Without them, two Claude runs produce different artifacts for
> the same install — that's a determinism + safety bug.

## What this prompt produces

For each of the four local artifact kinds, a template the LLM
fills in with slug-specific content:

- `local-cron` → bash script + crontab line
- `local-launchd` → launchd plist + bash script
- `local-gh-workflow` → workflow YAML
- `local-gstack-skill` → SKILL.md + optional supporting files

Each template includes:
- The literal scaffold (with `{{placeholder}}` slots)
- The variable list the LLM must fill in
- A safety checklist (env var assertions, error handling, etc.)
- The exact `installs.jsonl` row schema for this kind
- The exact uninstall steps

## Hard contracts (apply to all kinds)

- **No artifact gets written without `render-confirm-revoke`** —
  see recommend.md's "Local-install safety contract" section.
  The render in plain language + the literal artifact bytes both
  surface BEFORE any `Write` / `crontab -` / `launchctl load` /
  `chmod` happens. User must say "yes" / "install it" / "go ahead."
- **Every script begins with `set -euo pipefail`** (bash) or
  equivalent strict-mode header. Silent failures in scheduled
  scripts compound — strict mode surfaces them in stderr.
- **Required env vars are asserted at script-top** with the
  pattern `: "${VAR_NAME:?VAR_NAME must be set in shell env}"`.
  This makes the failure mode obvious if the user runs the
  script outside their normal shell (cron jobs don't inherit
  shell env reliably).
- **Every artifact carries a header comment** with: the slug,
  the generation date, the uninstall command. So a user later
  reading `~/.local/bin/` files knows where each came from.
- **No raw secrets in artifact content.** Read from env vars,
  read from `~/.murmur/account.json`, read from system keychains
  — never inline.
- **No network calls in the install step itself.** The artifact
  *runs* later and may make network calls. The install step is
  pure file I/O + crontab/launchctl. This makes installs
  airplane-friendly and reliably idempotent.

## Cron entry — `local-cron`

**Files written:**
- `~/.local/bin/mur-{{slug}}.sh` (the script body, `chmod 755`)
- One line in the user's crontab via
  `(crontab -l 2>/dev/null; echo "{{cron_expr}} ~/.local/bin/mur-{{slug}}.sh") | crontab -`

**Bash script scaffold:**

```bash
#!/usr/bin/env bash
# mur-{{slug}}.sh
# Generated by /mur recommend on {{iso_date}}
# To uninstall: tell Mur "uninstall {{slug}}" in chat — Mur runs
# the recorded uninstall_steps from ~/.murmur/installs.jsonl.
#
# What this does (one short paragraph): {{plain_english_summary}}
set -euo pipefail

# === Required env vars ===
# These must be set in the shell that runs the script. Cron jobs
# inherit env from /etc/environment + the user's per-cron-line env;
# if a var is missing, fail loud here rather than silently 4hr later.
{{#each required_env_vars}}
: "${{{this}}:?{{this}} must be set (paste into ~/.zshrc or run via a wrapper)}"
{{/each}}

# === State directory ===
STATE_DIR="${HOME}/.local/share/mur-{{slug}}"
mkdir -p "${STATE_DIR}"
LOG_FILE="${STATE_DIR}/run.log"

# === Lock to prevent overlap ===
LOCK_FILE="${STATE_DIR}/run.lock"
exec 9>"${LOCK_FILE}"
flock -n 9 || { echo "$(date -Iseconds) skipped (another run in progress)" >> "${LOG_FILE}"; exit 0; }

# === Main work ===
{{main_logic}}

# === Alert dispatcher ===
# Output goes to one of: macOS Notification Center, Slack webhook,
# email via mailx, GitHub issue. The persona's `alert_destination`
# field in the propose schema decides which.
{{alert_dispatcher}}

echo "$(date -Iseconds) ok" >> "${LOG_FILE}"
```

**Variables to fill:**
- `{{slug}}` — the install slug (matches `installs.jsonl` row)
- `{{iso_date}}` — `date -u +%Y-%m-%dT%H:%M:%SZ`
- `{{plain_english_summary}}` — one sentence, what this script does
- `{{required_env_vars}}` — array of env var names this script reads
- `{{cron_expr}}` — the cron expression (e.g. `0 */4 * * *`)
- `{{main_logic}}` — the bash body that does the actual work. Use
  `curl -fsSL` for HTTP calls (fail-fast), `jq` for JSON parsing.
  When calling APIs, prefer their idiomatic auth header pattern
  (Bearer for most modern APIs, Basic for Twilio, etc.).
- `{{alert_dispatcher}}` — one of:
  - **macOS notification:** `osascript -e 'display notification "{{message}}" with title "{{title}}"'`
  - **Slack webhook:** `curl -fsSL -X POST -H 'Content-Type: application/json' -d "{\"text\":\"{{message}}\"}" "${SLACK_WEBHOOK_URL}"`
  - **Email via mailx:** `echo "{{body}}" | mailx -s "{{subject}}" "${ALERT_EMAIL}"`
  - **GitHub issue:** `gh issue create --title "{{title}}" --body "{{body}}" --label "{{label}}"`

**`installs.jsonl` row:**
```json
{
  "ts": "<iso>",
  "event": "install",
  "slug": "{{slug}}",
  "kind": "local-cron",
  "artifact_path": "~/.local/bin/mur-{{slug}}.sh",
  "cron_line": "{{cron_expr}} ~/.local/bin/mur-{{slug}}.sh",
  "uninstall_steps": [
    "rm ~/.local/bin/mur-{{slug}}.sh",
    "crontab -l | grep -v 'mur-{{slug}}' | crontab -",
    "rm -rf ~/.local/share/mur-{{slug}}"
  ],
  "session_id": "<recommend session id>"
}
```

## Launchd plist — `local-launchd` (macOS only)

**Files written:**
- `~/.local/bin/mur-{{slug}}.sh` (same script scaffold as cron above)
- `~/Library/LaunchAgents/dev.usemur.{{slug}}.plist` (the plist)
- Loaded via `launchctl load ~/Library/LaunchAgents/dev.usemur.{{slug}}.plist`

**Plist scaffold:**

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>Label</key>
  <string>dev.usemur.{{slug}}</string>
  <key>ProgramArguments</key>
  <array>
    <string>/bin/bash</string>
    <string>-c</string>
    <string>{{absolute_script_path}}</string>
  </array>
  <key>StartInterval</key>
  <integer>{{interval_seconds}}</integer>
  <key>RunAtLoad</key>
  <true/>
  <key>StandardOutPath</key>
  <string>{{home}}/.local/share/mur-{{slug}}/stdout.log</string>
  <key>StandardErrorPath</key>
  <string>{{home}}/.local/share/mur-{{slug}}/stderr.log</string>
  <key>EnvironmentVariables</key>
  <dict>
    <key>PATH</key>
    <string>/usr/local/bin:/opt/homebrew/bin:/usr/bin:/bin</string>
    {{#each preserved_env_vars}}
    <key>{{this}}</key>
    <string>${{ this }}</string>
    {{/each}}
  </dict>
</dict>
</plist>
```

**Variables to fill:**
- `{{slug}}` — install slug
- `{{absolute_script_path}}` — full expanded path
  (`/Users/<user>/.local/bin/mur-<slug>.sh`). Don't use `~`; launchd
  doesn't expand it.
- `{{interval_seconds}}` — schedule in seconds (3600 = hourly,
  14400 = 4-hourly, 86400 = daily). For specific time-of-day, use
  `StartCalendarInterval` (dict with Hour/Minute keys) instead of
  `StartInterval`.
- `{{home}}` — `$HOME` expanded (full path)
- `{{preserved_env_vars}}` — env vars the script needs that
  launchd should pass through. List explicitly; launchd doesn't
  inherit the shell's full env.

**`installs.jsonl` row:**
```json
{
  "ts": "<iso>",
  "event": "install",
  "slug": "{{slug}}",
  "kind": "local-launchd",
  "artifact_path": "~/Library/LaunchAgents/dev.usemur.{{slug}}.plist",
  "script_path": "~/.local/bin/mur-{{slug}}.sh",
  "uninstall_steps": [
    "launchctl unload ~/Library/LaunchAgents/dev.usemur.{{slug}}.plist",
    "rm ~/Library/LaunchAgents/dev.usemur.{{slug}}.plist",
    "rm ~/.local/bin/mur-{{slug}}.sh",
    "rm -rf ~/.local/share/mur-{{slug}}"
  ],
  "session_id": "<recommend session id>"
}
```

**macOS-only.** On Linux, fall back to cron. Don't emit a launchd
plist if `uname` returns anything other than `Darwin`.

## GitHub Actions workflow — `local-gh-workflow`

**Files written:**
- `<project>/.github/workflows/mur-{{slug}}.yml` (the workflow)
- **Don't auto-commit.** Leave the file uncommitted; ask the user
  to commit + push themselves. The workflow lives in their repo,
  their control. Tell them: "Commit and push to enable."

**Workflow scaffold:**

```yaml
# .github/workflows/mur-{{slug}}.yml
# Generated by /mur recommend on {{iso_date}}
# To uninstall: tell Mur "uninstall {{slug}}" in chat (Mur removes
# the file + reminds you to commit), or rm this file + commit
# yourself.

name: mur-{{slug}}
on:
  schedule:
    - cron: '{{cron_expr}}'
  workflow_dispatch: {}

permissions:
  contents: read
  issues: write
  {{#each extra_permissions}}
  {{this.scope}}: {{this.level}}
  {{/each}}

jobs:
  run:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run watcher
        env:
          {{#each required_secrets}}
          {{this}}: ${{ secrets.{{this}} }}
          {{/each}}
        run: |
          set -euo pipefail
          {{main_logic}}
```

**Variables to fill:**
- `{{slug}}`, `{{iso_date}}` — same as cron
- `{{cron_expr}}` — GitHub Actions cron syntax (UTC; document the
  conversion in the rendered plain-English summary)
- `{{extra_permissions}}` — additional `permissions:` keys if the
  workflow writes to PRs, releases, etc.
- `{{required_secrets}}` — array of GH repo secret names
  (`STRIPE_API_KEY`, `SLACK_WEBHOOK_URL`, etc.). User must add
  these in the repo's Settings → Secrets. Render that requirement
  in the install confirm step.
- `{{main_logic}}` — the bash body. Same shape as cron.

**`installs.jsonl` row:**
```json
{
  "ts": "<iso>",
  "event": "install",
  "slug": "{{slug}}",
  "kind": "local-gh-workflow",
  "artifact_path": "<project>/.github/workflows/mur-{{slug}}.yml",
  "uninstall_steps": [
    "rm <project>/.github/workflows/mur-{{slug}}.yml",
    "git add -A && git commit -m 'remove mur-{{slug}} workflow'",
    "git push"
  ],
  "uncommitted_at_install": true,
  "required_secrets": [{{required_secrets}}],
  "session_id": "<recommend session id>"
}
```

**Note: `uncommitted_at_install: true` flag.** Until the user
commits + pushes, the workflow doesn't run on GH's schedule. When
the user asks Mur "show installs" / "what did you install"
(uninstall.md list mode), this row surfaces with an "(awaiting
commit + push)" badge so the user knows it's not yet active.

## gstack skill — `local-gstack-skill`

**Files written:**
- `~/.claude/skills/{{slug}}/SKILL.md` (the skill body)
- Optional supporting files (templates, scripts) in the same dir

**SKILL.md scaffold:**

```markdown
---
name: {{slug}}
description: {{one_line_description}}
---

# {{title}}

> Generated by /mur recommend on {{iso_date}}
> To uninstall: tell Mur "uninstall {{slug}}" in chat (or
> rm -rf ~/.claude/skills/{{slug}}/ if you'd rather do it
> yourself).

## When to use

{{when_to_use_paragraph}}

## What this skill does

{{behavior_description}}

## Activation phrases

{{activation_phrases_list}}

## Hard contracts

{{contracts_specific_to_this_skill}}
```

**Variables to fill:**
- `{{slug}}`, `{{iso_date}}` — same as cron
- `{{title}}` — human-readable name (Title Case)
- `{{one_line_description}}` — for the skill frontmatter; this is
  what Claude reads when deciding to activate
- `{{when_to_use_paragraph}}` — chief-of-staff voice, ~3 sentences
- `{{behavior_description}}` — what the skill does step-by-step
- `{{activation_phrases_list}}` — bullet list of phrases that
  trigger the skill
- `{{contracts_specific_to_this_skill}}` — any guarantees the
  skill makes (e.g. "never writes to user files," "always asks
  before paid call")

**`installs.jsonl` row:**
```json
{
  "ts": "<iso>",
  "event": "install",
  "slug": "{{slug}}",
  "kind": "local-gstack-skill",
  "artifact_path": "~/.claude/skills/{{slug}}/SKILL.md",
  "uninstall_steps": [
    "rm -rf ~/.claude/skills/{{slug}}/"
  ],
  "session_id": "<recommend session id>"
}
```

## Render order during install (mandatory)

1. **Plain-language summary** of what the artifact does. Three to
   five lines. Reference the persona's `why-you` from the propose
   schema.
2. **Where it lives** — file paths, sizes (line counts).
3. **How to undo** — render as a chief-of-staff sentence the
   user actually reads, not as a typed slash command. E.g.
   "Say 'uninstall {{slug}}' anytime — I'll remove the script,
   drop the cron line, and append an `uninstalled` audit row."
   Never render `/mur uninstall {{slug}}` as a typed instruction;
   `/mur` isn't registered with Claude Code and would error.
4. **The literal artifact bytes** — the rendered scaffold. Show
   in full when the artifact is short (cron entry, gstack skill
   <100 lines). For long artifacts, show the header + a
   representative excerpt + offer "say 'show me the full body' for
   the rest."
5. **Confirm prompt** — `Proceed? (yes / no / show me the full
   body / edit <field>)`.

Write to disk only after explicit "yes." Bare silence → cancel.
"Yes" alone after a different prompt → ambiguous → re-ask.
